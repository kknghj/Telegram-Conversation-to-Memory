# Future Roadmap

확장의 기준은 기술적 흥미가 아니라 **원문 충실한 기억 아카이브가 실제로 다시 읽을 가치가 있는가**입니다.

## Phase 0 - 현재 MVP 안정화와 검증

**목적**: 기능을 더 늘리기 전에 기록 품질과 사용자 신뢰를 확인

- 최신 스키마의 기억 30~50개 축적
- 1~2주 뒤 원문 충실도와 회고 가치 평가
- 저장 전 수정 사유 수집
- 후속 질문이 필요했던 비율과 중단율 확인
- active/cancelled/saved 초안 상태 전환 안정화
- README, 제품 원칙, 코드 동작의 지속적 일치

**진입 조건**: 없음  
**완료 조건**: 데이터 손실 없이 반복 사용 가능하고, 회고 가치가 확인됨

## Phase 1 - 백업 가능한 영속 저장

**목적**: 로컬 파일 의존과 단일 기기 운영 한계 해결

- Supabase `memories`와 `drafts` 테이블 설계
- 현재 JSON 스키마를 JSONB payload로 유지
- `MemoryStorage` 인터페이스를 구현하는 `SupabaseStorage` 추가
- 사용자별 Row Level Security
- 로컬 JSON 가져오기와 백업 경로 제공

이 단계는 로컬 데이터 손실 위험이나 멀티디바이스 필요가 실제로 나타날 때 진행합니다.

## Phase 2 - 기본 회고와 필터

**목적**: AI 의미 검색 전에 구조화 필드만으로 회고 가치 검증

- 날짜 범위 조회
- 사람, 프로젝트, 태그, 감정 필터
- 최근 기록 목록
- 원문과 구조화 결과 함께 보기
- 수정 이력 또는 원문 대비 변경점 확인

먼저 설명 가능한 필터를 사용하고, Vector DB는 필터로 해결되지 않는 검색 수요가 확인된 뒤 도입합니다.

## Phase 3 - 의미 검색

**목적**: 표현은 다르지만 맥락이 비슷한 기억 탐색

- `event_summary`와 `memory_candidate` 임베딩
- pgvector 우선 검토
- 시간·사람·프로젝트 필터와 의미 검색 결합
- 검색 결과에 원문 근거와 해석 위험도 표시
- `/검색 <질문>` 또는 자연어 검색 명령

검색 결과가 새로운 해석을 사실처럼 제시하지 않도록 원문 링크와 근거를 유지합니다.

## Phase 4 - 사용자 주도 회고

**목적**: 자동 조언이 아니라 저장된 기억을 다시 보는 도구 제공

- 주간·월간 기록 묶음
- 반복 주제, 사람, 프로젝트, 감정의 빈도
- 사용자가 선택한 기억만 회고에 포함
- AI 요약과 원문 사실을 구분해 표시
- 자동 푸시보다 요청 기반 회고 우선

회고에서도 성장 서사나 교훈을 자동 생성하지 않는 현재 제품 원칙을 유지합니다.

## Phase 5 - 외부 도구 연동

### Notion

- 승인된 기억만 데이터베이스에 동기화
- topic, user_emotions, people, projects, tags 매핑
- 원문 대화는 페이지 본문 또는 연결 페이지에 보관

### Hermes

- 외부 이벤트를 기록 시작 계기로 연결
- 승인된 기억을 노트나 워크플로 입력으로 전달
- 외부 시스템이 사용자 승인 단계를 우회하지 못하도록 제한

## 우선순위

```text
현재 MVP 안정화
    ↓ 기억 30~50개와 회고 가치 확인
백업 가능한 영속 저장
    ↓ 실제 조회 요구 수집
기본 필터와 회고
    ↓ 필터로 해결되지 않는 검색 수요 확인
의미 검색
    ↓ 사용자가 요청하는 회고 패턴 확인
사용자 주도 회고
    ↓
Notion / Hermes 연동
```

## 유지할 불변 원칙

확장 이후에도 다음은 바꾸지 않습니다.

- 사용자의 원문이 최우선 근거다.
- AI 해석은 사실과 구분한다.
- 후속 질문은 기록 정확도를 위해서만 사용한다.
- 최종 저장에는 사용자의 명시적 승인이 필요하다.
- 취소는 즉시 삭제와 같은 뜻이 아니다.
- 검색과 회고 결과에서도 원문으로 돌아갈 수 있어야 한다.

## 확장 포인트

| 위치 | 확장 방향 |
|------|-----------|
| `storage/local_json.py` | Supabase 저장 구현 추가 |
| `memory/service.py` | 임베딩과 회고용 별도 서비스 분리 |
| `memory/fidelity.py` | 한국어 과해석 패턴과 근거 검사 강화 |
| `bot/handlers.py` | 조회·검색·회고 명령 추가 |
| JSON schema | `embedding_id`, `source`, `revision`, `external_refs` 추가 |
| `app/database.py` | 최종 기억과 세션 테이블 실제 사용 |
