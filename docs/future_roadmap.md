이 로드맵의 기준은 기능을 많이 붙이는 것이 아니라, **신뢰할 수 있는 기억 기질에서 출발해 사용자가 읽고 싶어지는 콘텐츠를 스스로 준비하는 개인 AI 에이전트로 발전하는 것**입니다.

현재 프로젝트는 기억 아카이브 MVP입니다. 장기 목표는 기억 기반 콘텐츠 발행 에이전트입니다. 따라서 로드맵은 저장소 확장, 검색 기능, 회고 기능을 각각 따로 추가하는 목록이 아니라 다음 발전 과정으로 이해해야 합니다.

```text
원문 충실한 기억 저장
  → 장기 기억 기질 구축
  → 회고 씨앗과 연결 후보 축적
  → 역할 분리형 콘텐츠 발견/편집
  → 발행 판단과 알림 경험
  → 피드백 기반 품질 개선
```

## Phase 0 - 현재 MVP 안정화: 신뢰 가능한 기억 아카이브

**목적**: 장기 콘텐츠 생성의 재료가 될 기억을 왜곡 없이 축적한다.

현재 구현의 중심 단계입니다.

- Telegram 기록 흐름 안정화
- 원문 기반 JSON 구조화
- 감정 근거, 사람, 프로젝트, 태그, 핵심 표현 저장
- `model_interpretation`과 `memory_candidate` 분리
- `reflection_seed_candidate`, `open_questions`, `emerging_themes` 축적
- 저장 전 사용자 검토와 명시적 승인 유지
- 취소 초안 복구와 보관 정책 유지
- 원문에 없는 성장 서사, 긍정적 재해석, 코칭 톤 억제

**완료 조건**

- 데이터 손실 없이 반복 사용 가능하다.
- 승인된 기억 30-50개 이상이 쌓인다.
- 1-2주 뒤 다시 읽었을 때 원문 충실도와 회고 가치가 확인된다.
- 장기 분석에 쓸 수 있는 필드가 안정적으로 채워진다.

## Phase 1 - 기억 기질 고도화: 나중에 다시 읽을 수 있는 구조

**목적**: 단건 저장을 넘어, 시간이 지나도 연결하고 재해석할 수 있는 기억 기반을 만든다.

- Supabase 기반 영속 저장을 기본 운영 경로로 정리
- 승인 기억, 초안, 취소 기록, 수정 이력을 구분
- 기억 ID, 원문 참조, 생성 시점, 승인 상태, 해석 위험도 추적
- `reflection_seed`, `open_questions`, `value_tags`, `temporal_status` 같은 장기 분석 필드 안정화
- 기존 Local JSON 가져오기와 백업 경로 유지
- 저장소가 바뀌어도 원문 근거와 승인 흐름은 유지

**완료 조건**

- 기억이 파일 단위 보관을 넘어 조회, 연결, 재처리 가능한 상태가 된다.
- 장기 콘텐츠 후보가 특정 기억 ID와 원문 근거로 돌아갈 수 있다.

## Phase 2 - 회고 씨앗 수집: 사용자가 요청한 회고 전 단계

**목적**: 콘텐츠를 바로 쓰기 전에, 콘텐츠가 될 만한 재료를 더 잘 모은다.

현재 코드에는 이미 질문 분리와 회고 씨앗을 향한 초기 구조가 들어가 있습니다. 이 단계는 그것을 제품적으로 정돈합니다.

- 질문 생성과 요약 생성을 분리
- `meaning_check` 남용을 줄이고, association, contrast, memory_link 같은 질문 유형 사용
- 질문 세션 상태와 피로 신호 추적
- 기억마다 콘텐츠 재료로 쓸 수 있는 요소를 표시
  - 핵심 표현
  - 열린 질문
  - 반복 주제
  - 가치 판단
  - 장면과 감각
  - 프로젝트/관계 맥락
- 사용자가 회고를 요청하지 않아도 내부 후보로 쓸 수 있는 `reflection_seed`를 안정적으로 남김

**완료 조건**

- 저장된 기억이 단순 사건 요약이 아니라 콘텐츠 후보의 재료를 포함한다.
- 질문이 코칭처럼 느껴지지 않고, 사용자의 표현을 더 풍부하게 남기는 데 기여한다.

## Phase 3 - Reporter: 콘텐츠 후보 발견 시스템

**목적**: 기억에서 흥미로운 연결점과 콘텐츠 후보를 많이 발견한다.

Reporter는 정답을 내는 역할이 아닙니다. 많이 읽고, 많이 발견하고, 다양한 후보를 만드는 역할입니다.

- 최근 기억과 오래된 기억 사이의 연결 후보 탐색
- 반복 표현, 반복 감정, 반복 프로젝트, 반복 관계 맥락 발견
- `open_questions`가 다시 등장하는 순간 감지
- 특정 표현이나 장면이 다른 기억과 겹치는지 탐색
- 콘텐츠 후보를 `pitch` 형태로 저장
- 후보마다 근거 기억, 연결 유형, 흥미 포인트, 불확실성을 함께 기록

예시 후보:

- "사용자가 여러 프로젝트에서 '사람들의 시간을 줄이고 싶다'는 표현을 반복한다."
- "회의 기록과 강의 선택 기록이 모두 '불필요한 상주 시간'이라는 감각으로 연결된다."
- "최근 세 번의 기록에서 '내가 되고 싶은 인간상'이 다른 장면으로 나타난다."

**완료 조건**

- 사용자가 명시적으로 요청하지 않아도 내부 콘텐츠 후보가 쌓인다.
- 후보는 반드시 원문 근거와 연결되어 있다.
- 후보 생성은 넓고 다양하지만, 아직 사용자에게 바로 발행하지 않는다.

## Phase 4 - Editor: 근거 검토와 초안 숙성

**목적**: 발견된 후보를 그대로 발행하지 않고, 근거와 반례를 검토해 읽을 만한 초안으로 다듬는다.

Editor는 Reporter가 발견한 후보를 의심하고, 보완하고, 더 나은 표현으로 숙성시키는 역할입니다.

- 후보의 근거 기억이 충분한지 확인
- 반례가 되는 기억이나 상충되는 표현 탐색
- 단정적 해석, 과도한 심리 분석, 성장 서사 제거
- 콘텐츠 초안에서 사실, 인용, 해석을 분리
- 약한 후보는 보류하거나 폐기
- 여러 기억이 더 쌓일 때까지 대기시킬 후보 관리
- 하나의 후보를 짧은 글, 질문 모음, 장면 노트 등 여러 형식으로 시험

**완료 조건**

- 콘텐츠 초안마다 근거, 반례, 해석 위험도, 발행 준비 상태가 있다.
- 발행되지 않은 초안도 계속 수정되고 보류될 수 있다.
- 흥미롭지만 근거가 약한 글을 발행하지 않는 기준이 생긴다.

## Phase 5 - Editor-in-Chief: 발행 판단과 알림 경험

**목적**: 무엇을 언제 보여줄지 결정하고, 사용자가 부담보다 호기심을 느끼게 한다.

Editor-in-Chief는 콘텐츠 품질뿐 아니라 사용자 경험을 관리합니다.

- 발행 준비 완료 상태 정의
- 사용자에게 보낼 알림 문구와 빈도 관리
- 너무 비슷한 콘텐츠가 연속 발행되지 않도록 조절
- 민감하거나 해석 위험이 큰 콘텐츠는 발행 보류
- 사용자의 최근 활동, 읽기 반응, 피로도를 고려
- "새로운 콘텐츠가 준비되었습니다" 같은 낮은 압력의 알림 제공
- 사용자가 읽기, 나중에 보기, 숨기기, 수정, 피드백을 선택할 수 있게 함

**완료 조건**

- 콘텐츠는 생성 즉시 노출되지 않는다.
- 발행 타이밍과 빈도가 관리된다.
- 사용자는 알림을 프롬프트 요구가 아니라 읽을거리 제안으로 경험한다.

## Phase 6 - Feedback Steward: 발행 후 학습 루프

**목적**: 사용자의 반응을 다음 콘텐츠 품질에 반영한다.

- 읽음, 저장, 숨김, 수정, 직접 피드백 기록
- 사용자가 좋아하는 콘텐츠 형식과 싫어하는 해석 방식 구분
- 과해석 피드백을 Prompt Policy와 Editor 기준에 반영
- 자주 무시되는 후보 유형은 발행 우선순위 하향
- 사용자가 명시적으로 좋아한 연결 방식은 이후 후보 평가에 반영

**완료 조건**

- 발행 후 반응이 다음 후보 발견, 편집, 발행 판단에 영향을 준다.
- 피드백은 참여율 최적화가 아니라 신뢰도와 흥미도 개선에 사용된다.

## Phase 7 - 개인 콘텐츠 표면 확장

**목적**: Telegram 알림을 넘어, 사용자가 자신의 기억 기반 콘텐츠를 편하게 읽는 표면을 제공한다.

가능한 방향:

- Telegram 안에서 짧은 콘텐츠 카드 제공
- 웹 기반 개인 읽기 보관함
- 발행된 콘텐츠 아카이브
- 기억 원문과 콘텐츠 초안 사이의 근거 보기
- Notion 등 외부 도구로 승인된 콘텐츠만 내보내기
- 월간/계절별 선별 콘텐츠 모음

이 단계에서도 핵심은 대시보드가 아니라 읽기 경험입니다. 사용자가 관리해야 할 화면을 늘리는 것이 아니라, 준비된 콘텐츠를 편하게 읽게 하는 방향이어야 합니다.

## 유지할 불변 원칙

확장 이후에도 다음은 바꾸지 않습니다.

- 사용자의 원문이 최우선 근거다.
- AI 해석은 사실과 구분한다.
- 사용자가 승인한 기억과 내부 후보/초안은 상태를 분리한다.
- 콘텐츠는 생성 즉시 발행하지 않는다.
- 발행 판단에는 품질, 근거, 타이밍, 사용자 피로도를 함께 고려한다.
- 생산성, 코칭, 자기계발 조언이 아니라 흥미로운 읽기 경험을 목표로 한다.
- 사용자는 언제든 해석을 수정하거나 거부할 수 있다.

## 우선순위 요약

```text
1. 현재 MVP 안정화
   원문 충실한 기억과 회고 씨앗을 안정적으로 저장한다.

2. 기억 기질 고도화
   기억 ID, 원문 근거, 승인 상태, 장기 분석 필드를 정리한다.

3. 회고 씨앗 수집
   질문 분리와 reflection_seed 품질을 높인다.

4. Reporter 도입
   내부 콘텐츠 후보를 많이 발견한다.

5. Editor 도입
   후보의 근거와 반례를 검토하고 초안을 숙성한다.

6. Editor-in-Chief 도입
   발행 여부, 타이밍, 알림 경험을 관리한다.

7. Feedback Steward 도입
```
