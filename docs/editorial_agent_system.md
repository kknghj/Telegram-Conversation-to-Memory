# Editorial Agent System

## 목적

이 문서는 Conversation-to-Memory가 장기적으로 지향하는 **기억 기반 콘텐츠 발행 에이전트**의 역할 구조를 정의합니다.

현재 구현은 Telegram 기록을 원문에 충실한 기억으로 정리하고 저장하는 아카이브 시스템입니다. 이 문서는 그 위에 올라갈 미래 구조를 설명합니다. 따라서 현재 코드의 모듈 이름이나 즉시 구현할 클래스 설계가 아니라, 제품이 유지해야 할 역할과 책임의 경계를 다룹니다.

## 왜 역할을 나누는가

기억 기반 콘텐츠는 한 번의 모델 호출로 만들기 어렵습니다.

좋은 콘텐츠에는 서로 다른 판단이 필요합니다.

- 많이 발견하는 능력
- 근거를 의심하는 능력
- 표현을 다듬는 능력
- 지금 보여줄지 기다릴지 판단하는 능력
- 사용자의 피드백을 다음 기준에 반영하는 능력

이 책임들을 하나의 AI에게 모두 맡기면 다음 문제가 생깁니다.

- 흥미로운 연결을 발견하자마자 과감하게 단정한다.
- 근거가 약한 초안도 바로 사용자에게 보여준다.
- 알림 빈도와 사용자 피로도를 콘텐츠 품질과 분리해서 보지 못한다.
- 사용자의 피드백이 다음 발행 기준에 체계적으로 반영되지 않는다.

따라서 이 프로젝트는 편집국처럼 역할을 분리합니다.

## 역할 개요

| 역할 | 핵심 질문 | 주요 산출물 |
|------|-----------|-------------|
| Archivist | 이 기억은 원문에 충실하게 보존되었는가? | 승인 기억, 원문 근거, 구조화 필드 |
| Reporter | 여기서 흥미로운 연결 후보는 무엇인가? | 콘텐츠 후보, 연결 근거, 관찰 메모 |
| Editor | 이 후보는 근거가 충분하고 더 좋은 해석으로 다듬을 수 있는가? | 검토된 초안, 반례, 해석 위험도 |
| Editor-in-Chief | 이것을 지금 사용자에게 보여줘도 되는가? | 발행 결정, 알림, 보류/폐기 판단 |
| Feedback Steward | 사용자의 반응은 다음 기준을 어떻게 바꾸는가? | 피드백 신호, 선호/거부 패턴, 품질 기준 업데이트 |

역할 이름은 제품 설계를 위한 언어입니다. 실제 구현에서는 하나의 프로세스, 배치 작업, 큐 기반 워커, LLM 호출, 규칙 기반 필터, 데이터베이스 상태 전이 등으로 나뉠 수 있습니다.

## 역할별 책임

### 1. Archivist

Archivist는 현재 MVP와 가장 가까운 역할입니다.

책임:

- 사용자 원문을 보존한다.
- 기억 초안을 구조화한다.
- 감정, 사람, 프로젝트, 태그, 핵심 표현, 열린 질문을 정리한다.
- `memory_candidate`와 `model_interpretation`을 분리한다.
- 원문에 없는 성장 서사, 긍정적 재해석, 심리 단정을 억제한다.
- 사용자의 승인 전에는 최종 기억으로 확정하지 않는다.

Archivist의 산출물은 콘텐츠가 아니라 **신뢰 가능한 기억 재료**입니다.

### 2. Reporter

Reporter는 축적된 기억을 읽고 콘텐츠 후보를 발견합니다.

책임:

- 여러 기억 사이의 반복 표현을 찾는다.
- 프로젝트, 사람, 감정, 가치관의 느슨한 연결을 찾는다.
- `reflection_seed`, `open_questions`, `key_phrases`를 다시 읽는다.
- 최근 기억과 오래된 기억 사이의 흥미로운 대비를 찾는다.
- 하나의 정답으로 단정하지 않고 여러 후보를 만든다.
- 후보마다 근거 기억과 불확실성을 함께 남긴다.

Reporter는 많은 후보를 만들어도 됩니다. 다만 사용자에게 직접 발행하지 않습니다.

좋은 Reporter 후보 예시:

- "최근 세 기록에서 '시간을 줄이고 싶다'는 표현이 다른 맥락으로 반복된다."
- "업무 스트레스 기록과 앱 선택 기록이 모두 '불필요한 상주 시간'이라는 감각으로 연결된다."
- "사용자가 '되고 싶은 인간상'을 말할 때 평가, 편의성, 타인의 시간이라는 주제가 함께 나타난다."

나쁜 Reporter 후보 예시:

- "사용자는 생산성에 집착한다." 원문보다 강한 성격 단정
- "결국 성장하고 있다." 금지된 성장 서사
- "이 문제를 해결해야 한다." 코칭 또는 과제화

### 3. Editor

Editor는 Reporter의 후보를 검토하고 초안으로 숙성시킵니다.

책임:

- 후보의 근거 기억이 충분한지 확인한다.
- 반례가 되는 기억이 있는지 찾는다.
- 연결이 흥미로운지, 억지인지 구분한다.
- 사실, 인용, 해석을 분리한다.
- 원문에 없는 심리 단정과 성장 서사를 제거한다.
- 더 좋은 제목, 구조, 표현을 시도한다.
- 근거가 약하면 초안을 보류하거나 폐기한다.

Editor의 핵심 태도는 "멋진 글을 빨리 쓰자"가 아니라 "이 해석을 아직 믿어도 되는가"입니다.

### 4. Editor-in-Chief

Editor-in-Chief는 발행 여부와 사용자 경험을 관리합니다.

책임:

- 초안이 지금 발행할 만큼 준비되었는지 판단한다.
- 알림 빈도와 타이밍을 조절한다.
- 비슷한 주제가 연속으로 발행되지 않게 한다.
- 민감하거나 해석 위험이 높은 콘텐츠를 보류한다.
- 사용자가 최근에 많이 무시한 형식의 콘텐츠를 줄인다.
- 낮은 압력의 알림을 보낸다.
- 읽기, 나중에 보기, 숨기기, 수정, 피드백 경로를 제공한다.

Editor-in-Chief는 콘텐츠 품질뿐 아니라 **사용자가 이 에이전트를 부담스럽게 느끼지 않는가**를 판단해야 합니다.

### 5. Feedback Steward

Feedback Steward는 발행 이후의 반응을 다음 판단 기준으로 되돌립니다.

책임:

- 사용자가 읽었는지, 저장했는지, 숨겼는지, 수정했는지 기록한다.
- 사용자가 불편해한 해석 방식을 분류한다.
- 좋아한 콘텐츠 형식과 연결 방식을 찾는다.
- 피드백을 Reporter, Editor, Editor-in-Chief의 기준에 반영한다.
- 참여율만 높이는 방향으로 학습하지 않도록 제한한다.

피드백은 사용자를 더 오래 붙잡기 위한 수단이 아니라, 더 정확하고 더 흥미로운 읽기를 만들기 위한 재료입니다.

## 콘텐츠 생명주기

```text
approved_memory
  → discovery_candidate
  → pitch
  → draft
  → under_editorial_review
  → ready_to_publish
  → published
  → feedback_collected
```

보류와 폐기도 정상 상태입니다.

```text
pitch
  → parked
  → revived_later

under_editorial_review
  → rejected

ready_to_publish
  → postponed
```

### 상태 설명

| 상태 | 의미 |
|------|------|
| `approved_memory` | 사용자가 승인한 기억. 모든 콘텐츠의 최상위 근거. |
| `discovery_candidate` | Reporter가 발견한 낮은 확신의 연결 후보. |
| `pitch` | 콘텐츠가 될 수 있도록 제목, 흥미 포인트, 근거가 정리된 후보. |
| `draft` | 사용자에게 보여줄 수 있는 글의 초기 형태. 아직 발행 금지. |
| `under_editorial_review` | Editor가 근거, 반례, 표현, 위험도를 검토하는 상태. |
| `ready_to_publish` | Editor-in-Chief가 발행 판단을 할 수 있는 상태. |
| `published` | 사용자에게 알림 또는 읽기 표면으로 노출된 상태. |
| `feedback_collected` | 발행 후 반응이 기록되어 다음 기준에 반영된 상태. |
| `parked` | 흥미롭지만 근거가 부족해 보류된 상태. |
| `rejected` | 과해석, 중복, 낮은 가치 등으로 폐기된 상태. |
| `postponed` | 콘텐츠는 준비됐지만 타이밍상 발행을 미룬 상태. |

## 콘텐츠 후보의 최소 정보

구현 방식과 무관하게, 콘텐츠 후보는 최소한 다음 정보를 가져야 합니다.

| 필드 | 목적 |
|------|------|
| `candidate_id` | 후보 식별자 |
| `source_memory_ids` | 근거가 되는 승인 기억 목록 |
| `candidate_type` | 반복 표현, 가치관, 장면, 관계, 프로젝트, 열린 질문 등 |
| `pitch` | 왜 흥미로운지 한두 문장으로 설명 |
| `evidence` | 원문 인용 또는 기억 필드 기반 근거 |
| `counter_evidence` | 반례 또는 조심해야 할 기억 |
| `interpretation_risk` | low, medium, high |
| `status` | 생명주기 상태 |
| `last_reviewed_at` | 마지막 검토 시점 |
| `publication_readiness` | 발행 준비도 |
| `user_feedback` | 발행 후 반응 요약 |

이 정보가 있어야 후보가 즉흥 생성물로 사라지지 않고, 시간이 지나며 숙성될 수 있습니다.

## 발행 콘텐츠의 원칙

발행 콘텐츠는 다음을 지켜야 합니다.

- 원문 인용 또는 기억 근거로 돌아갈 수 있어야 한다.
- 사용자가 말하지 않은 사실을 사실처럼 쓰지 않는다.
- 해석은 해석으로 표시한다.
- 조언, 처방, 자기계발 과제를 만들지 않는다.
- 민감한 기억은 더 높은 발행 기준을 적용한다.
- 같은 주제를 반복 발행해 사용자를 피로하게 만들지 않는다.
- 사용자가 수정하거나 거부할 수 있는 경로를 제공한다.

## 알림 경험

알림은 명령이나 과제가 아니라 읽을거리 초대여야 합니다.

좋은 알림:

- "새로운 콘텐츠가 준비되었습니다."
- "최근 기록에서 이어지는 짧은 글이 준비되었습니다."
- "전에 남긴 질문과 연결되는 메모가 하나 생겼습니다."

피해야 할 알림:

- "이번 주 회고를 완료하세요."
- "당신의 문제 패턴을 분석했습니다."
- "성장을 위해 읽어야 할 글이 있습니다."

알림 문구는 낮은 압력, 낮은 단정, 높은 호기심을 목표로 합니다.

## 현재 문서와의 관계

- `vision.md`: 이 시스템이 왜 필요한지 설명하는 최상위 제품 철학.
- `future_roadmap.md`: 현재 MVP에서 이 시스템으로 가는 단계적 경로.
- `architecture.md`: 현재 Telegram 기반 기억 아카이브의 실제 구조.
- `reflection_agent_change_plan.md`: 질문 분리와 회고 씨앗 수집을 위한 중간 단계 설계.
- `question_strategy.md`: Reporter 이전 단계에서 좋은 기억 재료를 남기기 위한 질문 전략.
- `PROMPT_POLICY.md`: Archivist와 Editor가 공유해야 할 원문 충실도 기준.
